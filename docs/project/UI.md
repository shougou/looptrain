整体判断：**现在已经有游戏气质了，但 UX 层级还不稳定。**
最大问题不是美术，而是**信息层、立绘层、对话层、操作层互相抢空间**。玩家刚进来会知道“这是一个剧情游戏”，但不一定能快速判断：

> 我现在在哪？
> 当前目标是什么？
> 我正在和谁对话？
> 我接下来应该点哪里？
> 哪些内容是系统反馈，哪些是角色对白？

这正好和你现在要做的 **助手 runtime、记忆 runtime、GoalEngine、指令系统** 有关。UI 必须服务这套系统闭环，而不是单纯展示剧情。

---

# 一、当前界面优点

先说可以保留的部分。

## 1. 氛围是对的

深色背景、金色重点字、列车场景、小宁立绘，已经有“冷峻悬疑 + 时间循环”的感觉。这个方向可以继续。

## 2. 顶部状态栏有价值

现在顶部显示：

```text
第 1 轮 · 08:45
第七节车厢 · AP 10
```

这个很好。LT 的核心就是循环、时间、位置、行动点数，这些信息必须常驻。

## 3. “线索 / 人物 / 记忆 / 帮助”四个入口方向正确

这四个入口刚好对应 LT 的核心系统：

```text
线索 = Clue Memory
人物 = NPC State
记忆 = Loop Memory
帮助 = Assistant Runtime
```

这个设计方向是对的。

## 4. 底部“对话 / 行动”切换是正确的

这是 LT 区别普通 AVG 的关键。
玩家不是只看剧情，而是可以主动行动。这个结构必须保留。

---

# 二、当前最大问题

## 问题 1：小宁立绘严重遮挡核心内容

现在小宁立绘位于页面中上部，直接盖住了：

* 场景描述；
* 当前目标；
* 行动按钮；
* 主要阅读区域。

这会造成一个问题：

> 立绘本来应该增强沉浸感，但现在变成了信息障碍。

尤其是手机竖屏里，中间区域是最宝贵的交互区域，不能被大立绘长期占用。

**建议：立绘不要常态居中覆盖。**

立绘应该有三种状态：

```text
1. 场景状态：小头像 / 半身缩略图
2. 对话状态：左侧或右侧半身立绘，不遮挡对白
3. 角色详情状态：点击人物后全屏展示大立绘
```

当前这种“悬浮大立绘压在内容上”的方式，只适合角色登场瞬间，不适合常态 UI。

---

## 问题 2：信息层级太多，玩家第一眼不知道看哪里

当前界面同时出现：

```text
顶部状态
当前目标
四个功能按钮
场景标题
场景描述
当前目标二次展示
行动按钮
对话记录按钮
大立绘
对话气泡
模式切换
输入框
发送按钮
```

这些信息都重要，但同时出现会让玩家失去主线。

LT 的首屏应该只回答四件事：

```text
现在是什么状态？
当前目标是什么？
我看到/听到了什么？
我可以做什么？
```

其他内容应该折叠或延后展示。

---

## 问题 3：“当前目标”重复且不够结构化

截图里顶部有：

```text
证明第七节车厢存在异常，并说服赵乘警检查地板。
```

中间又出现：

```text
当前目标：
……
```

目标信息重复出现，但没有形成任务进展感。

建议改成 GoalEngine 驱动的结构：

```text
当前目标：证明第七节车厢存在异常

进展：
✓ 小宁听到地板下方有声音
○ 找到可疑地板
○ 说服赵乘警检查

建议行动：
[追问小宁] [检查地板] [找赵乘警]
```

这样玩家能明确知道：

> 我已经完成了什么，还缺什么。

这比单句目标更有正反馈。

---

## 问题 4：对话区和场景区混在一起

现在上半部分是场景，下半部分是对话，但小宁立绘又覆盖在中间，导致玩家会不确定：

> 我现在是在探索场景，还是正在和小宁对话？

LT 需要明确区分两个状态：

```text
探索状态：显示场景描述 + 当前目标 + 推荐行动
对话状态：显示 NPC 信息 + 对话记录 + 对话输入
```

不要两个状态混在同一屏里抢空间。

---

## 问题 5：操作按钮的含义还不够清楚

比如：

```text
小宁 / 检查 / 前往过道处
```

这三个按钮混合了不同类型：

```text
小宁 = 人物交互
检查 = 行动
前往过道处 = 移动
```

对玩家来说可以点，但对系统设计来说不清晰。

建议按钮显示时带类型感：

```text
[询问小宁]
[检查地板]
[前往过道]
```

或者更明确：

```text
人物：小宁
行动：检查地板
移动：前往过道
```

试玩版阶段不要追求简短，优先让玩家理解系统边界。

---

# 三、推荐的新界面结构

我建议把界面重构成四层。

```text
1. 顶部状态层
2. 当前目标层
3. 主内容层
4. 底部指令层
```

---

## 1. 顶部状态层

保留，但压缩高度。

```text
第 1 轮 · 08:45        AP 10
第七节车厢             🔊
```

不要在顶部放太长目标句。
顶部只放硬状态：

```text
loopId
time
location
AP
sound
```

这些来自 RuntimeState。

---

## 2. 当前目标层

放在顶部状态下面，作为常驻 Goal Bar。

建议样式：

```text
当前目标
证明第七节车厢存在异常

进展：1 / 3
✓ 小宁听到异常
○ 找到异常位置
○ 说服赵乘警检查
```

如果空间紧，可以默认折叠：

```text
目标：证明车厢异常  进展 1/3  ˅
```

点击后展开详细进展。

这部分由 GoalEngine 输出，不要手写 UI 文案。

---

## 3. 主内容层

主内容层根据状态切换。

### 探索状态

展示场景：

```text
第七节车厢

车厢灯光昏暗。小宁抱着旧布娃娃坐在靠窗的位置。
她一直看向地板，像是在害怕那里有什么东西。

可互动对象：
小宁 / 地板 / 过道 / 乘客座位
```

此时立绘可以小尺寸显示在右侧或卡片中，不要遮挡正文。

---

### 对话状态

当玩家点击“小宁”后，主内容层切换成对话界面：

```text
小宁
状态：紧张 / 信任低 / 可能知道异常

[小宁小头像或半身立绘]

小宁：
“你……也听见了吗？”

你：
“你好。”

小宁：
“你也是去重庆的吗？不过现在只能往反方向走了。”
```

大立绘可以放在对话头部，但高度控制在 180–220px，不要覆盖对白区。

更好的方式是：

```text
左侧/上方：小宁头像或半身
右侧/下方：对白
```

而不是悬浮在正文中间。

---

## 4. 底部指令层

底部是 LT 的核心操作区，建议保持常驻。

结构：

```text
[对话] [行动]

输入框：和小宁说些什么……
[发送]
```

但建议增加“推荐指令 chips”，放在输入框上方：

```text
建议：
[问她听到了什么]
[问她为什么害怕地板]
[安抚她]
```

这对新手非常关键。

---

# 四、对话界面具体优化

## 当前对话界面的问题

现在对话已经出现气泡，但空间利用不理想：

* 角色名“小宁”离对白区太远；
* “结束对话”按钮位置偏上，不够明显；
* 玩家输入气泡和 NPC 气泡样式差异不够强；
* 底部输入区过重，占用空间较大；
* 大立绘让对话区被迫下移。

---

## 建议对话状态布局

可以改成：

```text
顶部状态栏
目标栏

NPC 对话头部：
[小宁头像] 小宁
状态：紧张 / 可追问

对话记录：
小宁：“你……也听见了吗？”
你：“你好。”
小宁：“你也是去重庆的吗？不过现在只能往反方向走了。”

推荐追问：
[你听见了什么？]
[你刚才看到了谁？]
[你为什么害怕地板？]

底部：
[对话] [行动]
输入框 + 发送
```

大立绘只在 NPC 详情页或首次登场时放大。

# 五、立绘使用建议

现在小宁立绘质量不错，但使用方式需要调整。

## 立绘三档显示

### 1. 小头像

用于：

```text
对话列表
人物入口
目标提示
```

尺寸：

```text
48px ~ 64px
```

---

### 2. 半身立绘

用于：

```text
进入对话状态
角色首次登场
关键剧情
```

尺寸：

```text
高度 180px ~ 240px
```

位置建议：

```text
主内容卡片右侧
或对白区上方居中
```

不能遮挡正文和按钮。

---

### 3. 全身大图

用于：

```text
人物详情页
角色卡
关键演出
```

当前这种大图适合人物详情页，而不是常态对话页。

# 六、GoalEngine 在 UI 中的呈现方式

你现在已经有 GoalEngine，因此 UI 不应该只显示“任务文字”，而要显示结构化目标。

建议 Goal ViewModel：

```ts
type GoalViewModel = {
  title: string
  description: string
  progress: {
    label: string
    done: boolean
  }[]
  suggestedActions: {
    label: string
    actionType: string
    payload: unknown
  }[]
}
```

UI 展示：

```text
当前目标：证明第七节车厢存在异常

✓ 已获得小宁证词
○ 尚未检查地板
○ 尚未说服赵乘警

建议行动：
[追问小宁] [检查地板] [找赵乘警]
```

这样玩家可以明确知道目标完成标准。

# 七、线索 / 人物 / 记忆 / 帮助 四个入口优化

这四个按钮现在比较像功能菜单，但可以更游戏化。

## 线索

显示已获得线索：

```text
已知线索 2

✓ 小宁听到地板下有金属声
✓ 第七节车厢地板存在异常震动
```

新线索出现时按钮角标：

```text
线索 +1
```

---

## 人物

显示人物状态：

```text
小宁
状态：紧张
信任：低
已知：听到异常声音

赵乘警
状态：巡逻中
可行动：需要证据才能说服
```

---

## 记忆

显示跨轮记忆：

```text
上一轮结论：
爆炸发生在 08:58。
第七节车厢异常最明显。
小宁可能是关键目击者。
```

---

## 帮助

不是普通帮助，而是许知微：

```text
许知微建议：
“你已经知道异常在第七节车厢，但还缺可以说服赵乘警的证据。先检查地板。”
```

按钮名称可以考虑改为：

```text
许知微
```

比“帮助”更符合剧情。

# 九、视觉风格建议

## 1. 主色可以继续用暗金色

暗金色适合 1939、旧列车、悬疑感。
但要控制使用位置。

建议：

```text
金色：目标、关键线索、按钮高亮
蓝色：记忆、系统提示
红色：危险、时间逼近
灰白：正文
```

不要所有重点都用金色，否则层级会混乱。

---

## 2. 卡片边框可以减少

现在卡片边框较多，容易显得“系统面板化”。
LT 应该保留系统感，但不要像后台管理页。

建议：

```text
主内容卡片：保留
小按钮卡片：弱化边框
目标栏：使用左侧竖线或浅背景
对话气泡：保留圆角
```

---

## 3. 字号层级建议

手机端建议：

```text
顶部状态：13px
目标标题：15px / 加粗
场景标题：22px
正文：16px ~ 17px
对白：16px
按钮：15px
```

当前正文偏大，标题也偏大，导致空间紧张。

# 十、交互优先级建议

从开发角度，不要一次性重做全部。建议按优先级处理。

## P0：必须马上改

```text
1. 立绘不能遮挡正文和按钮
2. 当前目标改成结构化进展
3. 探索状态和对话状态分离
4. 推荐行动按钮文案改成明确动作
```

这四个会直接影响试玩体验。

---

## P1：0.7 应该完成

```text
1. 线索 +1 正反馈
2. 目标完成轻提示
3. 许知微主动建议
4. 记忆页显示上一轮摘要
5. 对话状态下显示 NPC 状态
```

这些能体现 runtime 系统价值。

---

## P2：后续增强

```text
1. 立绘呼吸 / 眨眼 / 轻微位移
2. 时间逼近时 UI 变暗或抖动
3. 关键线索出现时短动画
4. 目标完成音效
5. 失败复盘动画
```

这些是氛围增强，不是当前最关键的。

---

# 十一、核心改造原则

这版 UI 的目标不是“更漂亮”，而是让玩家理解 LT 的核心逻辑。

每一屏都应该明确表达：

```text
当前状态：我在哪一轮、几点、在哪
当前目标：系统希望我确认什么
当前进展：我已经知道什么，还缺什么
当前行动：我现在可以做什么
当前反馈：我的行动是否产生了线索、记忆或路线变化
```

如果 UI 能稳定回答这五个问题，试玩版体验会明显提升。

最终建议：

> **把大立绘从“中心遮挡层”降级为“角色状态层”；把当前目标从“提示语”升级为“结构化进度条”；把底部输入从“聊天框”升级为“对话 / 行动双模式指令入口”。**

这会更符合 LT 现在的系统进展，也能让玩家更快理解你背后的设计逻辑。
