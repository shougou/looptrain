## 草稿内容

### 背景

本次项目全面分析发现了一个根本问题：3 个检查脚本全部 PASS，但实际存在 23 处文档不一致。9 个版本号存储位置中 6 个版本号不一致（MANIFEST 停在 0.5，而实际已是 v0.8.1），但检查脚本只覆盖 3 个位置。问题的本质不是"开发者忘记更新文档"，而是"检查脚本覆盖范围 << 实际事实源分布"。

### 解决

从"依赖人工保持一致"转向"单一版本源 + 自动化同步 + 全覆盖检查"。

### 核心变更

1. **VERSION 文件** — 项目唯一版本源，所有 13 个位置从此派生
2. **sync_version.sh** — 自动同步 11 个派生位置，收尾阶段必须执行
3. **check_release_wrapup.sh 扩展** — 版本一致性从 4 个位置扩展到 13 个
4. **check_cross_consistency.py** — 新增 5 项跨文档检查（目录存在性/文件存在性/README一致性/changelog Removed验证/changelog Added验证）
5. **AGENT.md 规则 17-20** — 将以上所有变为强制可验证断言

### 数据

- 修复 23 处不一致（9 版本号 + 4 内容声明 + 7 文档结构 + 2 死代码 + 1 缺失文档）
- 16/16 AC 通过，0 ERROR
- 4 条新规则加入 Agent 开发协议

### 反思

工程规范的价值不在于文档本身，而在于让检查可验证、让流程可重复。本次扩展让"版本号一致"从一个人工检查项变成了自动可验证断言。核心原则不变：每一步有固定输入输出，每一条规定可验证，每一次收尾不留尾巴。
