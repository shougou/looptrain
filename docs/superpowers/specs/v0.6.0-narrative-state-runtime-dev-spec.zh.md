## 1. Version Positioning

### 1.1 v0.6.0 的唯一目标

v0.6.0 第一部分的目标是：

```text
把当前 SLT 的 flat snapshot state，升级为可测试、可迁移、可投影的 Narrative State Runtime Core。
```

它要解决的问题是：

```text
当前 state 只是存档快照，不是记忆系统。
```

当前 state 把以下不同生命周期的数据混在一个 JSON 里：

```text
loop
clock
ap_remaining
known_clues
carried_memory
npc_states
flags
dialogue_session
```

v0.6.0 要把它们拆分为结构化层：

```text
Player / Run / Loop
Event Log
Knowledge
Belief
Relationship
Timeline
Archive
Profile
Snapshot
PromptMemoryContext
```

### 1.2 与第二部分 Assistant Runtime 的关系

第二部分 `runtime-architecture-design.revised.zh.md` 负责：

```text
CompanionView
Deterministic Assistant Runtime
ActionPlanner
Assistant OutputValidator
LLMProvider
询问助手 UI
```

本文负责：

```text
Memory Runtime Core
Legacy snapshot migration
Narrative state model
PromptMemoryContext
Reset / Archive data semantics
```

两者依赖关系：

```text
v0.6.0 Narrative State Runtime
  ↓
PromptMemoryContext / future CompanionMemoryProjection
  ↓
CompanionView
  ↓
Deterministic Assistant Runtime
  ↓
LLM Expression Layer
```

v0.6.0 MUST NOT 反向依赖 Assistant Runtime。

### 3.3 Browser MemoryRuntime decision

长期设计中，MemoryRuntime canonical location 是 browser，因为最终 IndexedDB 由 browser 访问。

但 v0.6.0 第一部分 MUST NOT 引入 browser TypeScript bundling。

v0.6.0 第一部分的执行定位：

```text
server/test-side Runtime Core + legacy snapshot projection
```

即：

```text
TypeScript Runtime Core 编译为 CommonJS，供 server.js 和 node:test 使用。
current browser app.js 继续维护现有 legacy state。
server-side Runtime Core 通过 RuntimeClientState / LegacyStandaloneSnapshot 接收 browser state 的副本。
```

完整 browser MemoryRuntime / IndexedDB cutover 属于后续版本：

```text
v0.6.1 Browser MemoryRuntime + IndexedDB Host
```


```text

```

```json
{
  "scripts": {

    "test:runtime": "npm run build:runtime && node --test \"dist/runtime/**/*.test.js\"",
    "test:standalone": "node tests/smoke_test.js",
    "test": "npm run test:runtime && npm run test:standalone"
```

If existing `npm test` already exists, it MUST be preserved by integrating `test:runtime` rather than replacing all legacy tests.

## 6. Runtime Laws for v0.6.0

### Law 1: Engine is the only judge

Only `engine.js` / `LegacyEngineAdapter` may decide：

```text
action success/failure
AP cost
time advance
clue unlock
loop outcome
NPC state change
relationship change source
```

MemoryRuntime records and projects. It does not judge.

### Law 2: Event first, projection second

Memory state MUST be derived from events or migration input.

```text
Engine / Migration produces MemoryEventDraft[]
  ↓
MemoryRuntime appends MemoryEvent[]
  ↓
MemoryProjector updates derived records
```

### Law 3: Knowledge is confirmed; Belief is inferred

Belief MUST NOT be treated as Knowledge.

`PromptMemoryContextBuilder` MUST mark Belief as inferred and uncertain.

### Law 4: dialogue_session is temporary

`dialogue_session` MUST NOT be migrated as long-term memory.

It may appear only as：

```text
current EngineState / temporary runtime state
```

### Law 5: LLM does not update Memory

No LLM output may create or update：

```text
Knowledge
Belief
Relationship
Timeline
Archive
Profile
```

### Law 6: v0.6.0 must run without LLM

All v0.6.0 tests MUST pass with：

```text
LT_LLM_PROVIDER=disabled
```

### 7.3 Required generation rules

```text
playerId:   player_${uuidv4}

episodeId:  content-authored stable ID, v0.6.0 default trial-001
```

`eventSeq` MUST be strictly increasing inside one run.

## 8. Event Protocol

### 8.1 MemoryEventType

v0.6.0 MUST use UPPER_SNAKE_CASE event types only.

```ts
export type MemoryEventType =
  | 'PLAYER_CREATED'
  | 'RUN_STARTED'
  | 'LOOP_STARTED'
  | 'ACTION_COMMITTED'
  | 'AP_SPENT'
  | 'TIME_ADVANCED'
  | 'SCENE_CHANGED'
  | 'DIALOGUE_STARTED'
  | 'DIALOGUE_ENDED'
  | 'CLUE_UNLOCKED'
  | 'KNOWLEDGE_CONFIRMED'
  | 'BELIEF_CREATED'
  | 'BELIEF_UPDATED'
  | 'BELIEF_CONTRADICTED'
  | 'BELIEF_CONFIRMED'
  | 'RELATIONSHIP_UPDATED'
  | 'NPC_STATE_UPDATED'
  | 'TIMELINE_ENTRY_CREATED'
  | 'LOOP_FAILED'
  | 'LOOP_SUCCEEDED'
  | 'LOOP_OUTCOME_RECORDED'
  | 'ARCHIVE_ENTRY_CREATED'
  | 'CARRYOVER_MEMORY_RECORDED'
  | 'RESET_APPLIED'
  | 'SNAPSHOT_WRITTEN'
  | 'MIGRATION_APPLIED';
```

Lowercase event types are forbidden.

### 8.2 MemoryEventDraft

Server / migration code MUST produce drafts, not final events.

```ts
export interface MemoryEventDraft<TPayload = unknown> {
  type: MemoryEventType;
  runId: string;
  loopId: string;
  chapterId: string;
  episodeId: string;
  sceneId: string;
  clock?: string;
  source: 'engine' | 'migration' | 'reset' | 'runtime';
  targetId?: string;
  payload: TPayload;
}
```

### 8.3 MemoryEvent

MemoryRuntime append converts draft to final event.

```ts
export interface MemoryEvent<TPayload = unknown> extends MemoryEventDraft<TPayload> {
  eventId: string;
  eventSeq: number;
  playerId: string;
  prevEventId: string | null;
  createdAt: string;
}
```

`createdAt` MUST be ISO-8601 UTC string.

### 8.4 Append rule

Only `MemoryRuntime.appendDrafts()` may create final `MemoryEvent`.

`LegacyEngineAdapter` MUST NOT create eventId/eventSeq/prevEventId.

## 11. Storage Layer

### 11.1 Required v0.6.0 storage

v0.6.0 MUST implement：

```text
InMemoryMemoryStorage
SerializedMemoryStorage
```

v0.6.0 MAY define but MUST NOT require：

```text
IndexedDBMemoryStorage
LocalStorageMemoryStorage
```

Reason：v0.6.0 不引入 browser TypeScript bundle。完整 IndexedDB cutover 属于 v0.6.1。

### 11.2 MemoryStorageAdapter

```ts
export interface MemoryStorageAdapter {
  load(): Promise<MemoryState | null>;
  save(state: MemoryState): Promise<void>;
  clear(): Promise<void>;
}
```

v0.6.0 不实现复杂 multi-store transaction。复杂 IndexedDB transaction 在 v0.6.1 实现。

### 11.3 InMemoryMemoryStorage

Required behavior：

```text
load returns deep clone of state or null.
save stores deep clone.
clear removes state.
```

Tests MUST verify mutation isolation.

### 11.4 SerializedMemoryStorage

Purpose：

```text
Serialize / deserialize MemoryState for API bridge tests and future browser handoff.
```

It MUST reject：

```text
schemaVersion mismatch
missing player/run/currentLoop
invalid eventSeq
invalid event linkage
```

## 12. Legacy Snapshot Migration

### 12.1 LegacyStandaloneSnapshot

```ts
export interface LegacyStandaloneSnapshot {
  loop?: number;
  clock?: string;
  ap_remaining?: number;
  known_clues?: string[];
  carried_memory?: string[] | Record<string, unknown>;
  npc_states?: Record<string, unknown>;
  flags?: Record<string, unknown>;
  dialogue_session?: unknown;
  [key: string]: unknown;
}
```

### 12.2 Migrator interface

```ts
export interface LegacyStandaloneStateMigrator {
  migrate(input: LegacyStandaloneMigrationInput): LegacyStandaloneMigrationOutput;
}

export interface LegacyStandaloneMigrationInput {
  playerId?: string;
  runId?: string;
  chapterId: string;
  episodeId: string;
  sceneId: string;
  legacyState: LegacyStandaloneSnapshot;
  migratedAt: string;
}

export interface LegacyStandaloneMigrationOutput {
  state: MemoryState;
  eventDrafts: MemoryEventDraft[];
  warnings: string[];
}
```

### 12.3 Mapping rules

| Legacy field | v0.6.0 mapping |
|---|---|
| `loop` | `LoopRecord.loopIndex` |
| `clock` | event payload / PromptMemoryContext scene clock |
| `ap_remaining` | not Memory source of truth; included only in PromptMemoryContext scene snapshot |
| `known_clues` | `KnowledgeRecord[]` with `sourceType='migration'` |
| `carried_memory` | `KnowledgeRecord[]` + `ArchiveEntryRecord[]` if parseable |
| `npc_states` | `RelationshipRecord[]` |
| `flags` | engine internal flags; only safe visible flags may become event payload |
| `dialogue_session` | ignored; never migrated as long-term memory |

### 12.4 Required migration events

Migrator MUST produce at least：

```text
PLAYER_CREATED
RUN_STARTED
LOOP_STARTED
MIGRATION_APPLIED
```

For each migrated known clue, it SHOULD produce：

```text
KNOWLEDGE_CONFIRMED
```

For each migrated relationship, it SHOULD produce：

```text
RELATIONSHIP_UPDATED
```

### 12.5 Unknown legacy fields

Unknown fields MUST NOT fail migration.

They MUST be either：

```text
ignored with warning
or copied into safe migration diagnostics
```

They MUST NOT be added to Knowledge or Belief automatically.

## 14. MemoryProjector

### 14.1 Responsibility

MemoryProjector applies MemoryEvent to derived stores：

```text
Knowledge
Belief
Relationship
Timeline
Archive
Loop status
Snapshot metadata
```

It MUST NOT call Engine.

### 14.2 Projection rules

#### KNOWLEDGE_CONFIRMED

Payload：

```ts
export interface KnowledgeConfirmedPayload {
  clueId: string;
  title: string;
```

## 15. Reset Semantics

### 15.1 Soft Reset

Keeps：

```text
Knowledge
Belief
Relationship
Archive
Profile
```

Resets：

```text
current loop runtime pointer
clock/AP/dialogue_session are engine state, not MemoryRuntime state
```

Creates：

```text
RESET_APPLIED
LOOP_STARTED for next loop if requested by caller
```

### 15.2 Chapter Reset

Keeps：

```text
Profile
Archive
Knowledge where scope = meta
```

Forgets：

```text
Knowledge where scope = chapter/episode/loop
Belief
Relationship
Timeline for active run unless archived
```

### 15.3 Forget Reset

Keeps：

```text
Profile
Archive
```

Forgets：

```text
Knowledge
Belief
Relationship
Timeline
```

### 15.4 Developer Reset

Clears everything.

MUST be available only in tests/dev API.

## 22. Final Implementation Principle

v0.6.0 第一部分的最终原则：

```text
Engine judges.
Runtime records.
Memory projects.
Prompt Builder reads structure.
LLM performs only after structure is safe.
Assistant comes later.
```

不要先做许知微，不要先做提示按钮，不要先做 LLM 表达。

先让系统准确记录：

```text
玩家经历了什么；
玩家确认知道什么；
玩家相信但可能错误地相信什么；
每次循环留下了什么；
哪些记忆可以被带入下一轮；
哪些内容可以安全进入 Prompt。
```

这就是 v0.6.0 Narrative State Runtime 的完成标准。
