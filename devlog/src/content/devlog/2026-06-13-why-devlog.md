## 背景

历史背景：本文写于 v0.4.3 阶段，文中的 SillyTavern 描述属于早期验证架构，不代表当前 SLT 运行时。

LoopTrain 从最初的概念原型到现在的试玩版（v0.4.3），已经经历了多个技术方案和设计方向的调整。项目的核心基础设施（SillyTavern 引擎层 + LoopTrain 控制层）已经基本建立，但整个开发过程中的设计判断、技术决策和版本变化一直没有被系统性记录下来。

对于一个个人长期项目而言，口头记忆和本地笔记无法支撑持续 6 个月以上的开发节奏。每次重新打开项目时，需要花费大量时间回忆"上一次做到了哪里、为什么那样做、下一步应该做什么"。

因此决定建立 LoopTrain 开发日志网站，作为项目的公开记录站。

## 当前问题

在没有开发日志之前，项目存在以下问题：

1. **设计判断丢失**：做过的重要技术决策和权衡理由没有存档，后续难以回溯。
2. **版本状态模糊**：无法一眼看出当前项目处于什么阶段、存在哪些已知问题、下一步重点是什么。
3. **反馈入口缺失**：早期试玩用户不知道该项目的当前状态、已知限制和后续计划。
4. **长期节奏困难**：缺乏定期记录的习惯，导致项目推进节奏不稳定。

## 本次调整

建立开发日志网站的核心决策：

1. **采用静态站点方案**：使用 Astro + Markdown 构建，不引入数据库或复杂后台。内容以 Markdown 文件维护，版本控制在 Git 中。
2. **最小可用第一版**：首页、试玩入口、开发日志列表和详情、路线图、版本记录、关于页面。
3. **暗色克制视觉风格**：采用深色主题、冷蓝强调色、档案记录感，符合 LoopTrain 的列车悬疑氛围。
4. **移动端优先**：与 LoopTrain 游戏保持一致的移动端竖屏优先策略。

## 设计判断

为什么不使用传统的博客平台或 CMS？

- **长期可维护性**：Markdown 文件 + 静态生成 = 不依赖任何特定平台。10 年后如果还有人在维护这个项目，只需要一个文本编辑器就能更新。
- **内容控制**：不需要处理评论、用户系统、数据库维护等非核心问题。
- **归档质量**：静态站点的 URL 结构、页面布局、版本记录都是明确可预期的，适合长期归档。

为什么不做一个更"华丽"的游戏官网？

- 当前项目仍处于早期试玩阶段，目标是"记录"而非"宣传"。
- 华丽官网会制造不切实际的完成度预期，与项目的实际状态不符。
- 一个克制、真实的开发日志更容易建立信任。

## 后续计划

1. 完成首批开发日志（本文 + 问题整理 + 音效系统计划）和版本记录（v0.4.3）。
2. 建立路线图页面，明确当前阶段、下一阶段和长期方向。
3. 持续更新开发日志，每次版本变化或重要设计判断后记录。
