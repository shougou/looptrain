# AGENT.md

项目名称：LoopTrain Devlog / LoopTrain 开发日志  
项目类型：静态网站 / 开发日志站 / 独立游戏项目档案站  
目标域名：looptrain.me  
适用对象：AI Coding Agent、前端开发者、内容维护者  
文档版本：v0.1

---

## 1. 项目定位

LoopTrain Devlog 是 LoopTrain 游戏项目的开发日志网站。

该网站不是传统商业游戏官网，也不是社区平台，而是一个面向个人长期创作的静态内容站，用于记录：

- 游戏试玩入口
- 开发日志
- 版本更新记录
- 项目路线图
- 设计笔记
- 素材与授权记录
- 当前问题和后续计划

网站核心价值不是营销包装，而是让 LoopTrain 项目具备长期沉淀能力。

---

## 2. 项目铁律

所有开发、重构、设计和内容生成必须遵守以下铁律。

### 2.1 静态优先

本网站第一阶段必须保持静态站点形态。

禁止在第一阶段引入：

- 用户登录
- 数据库
- 评论区
- 论坛
- 复杂后台 CMS
- 付费系统
- 用户上传
- 服务端业务状态

允许使用：

- Markdown 内容管理
- 静态页面生成
- 静态 JSON 数据
- 静态资源
- Nginx 部署
- HTTPS

### 2.2 内容可长期维护

所有开发日志、设计笔记、更新记录应优先使用 Markdown 文件维护。

不允许把核心内容硬编码在组件里，除非该内容确实属于页面结构或设计常量。

推荐内容目录：

```text
src/content/
├── devlog/
├── design/
├── changelog/
└── assets/
```

### 2.3 不剧透核心悬疑

LoopTrain 是悬疑叙事解谜项目。

公开网站允许写：

- 设计方向
- 玩法机制
- 版本问题
- 技术实现
- 开发过程
- 非核心设定

禁止公开写：

- 最终真相
- 核心反转
- 关键角色隐藏身份
- 事故完整原因
- 关键谜题答案
- 尚未在游戏中公开的核心线索

### 2.4 不把计划写成承诺

Roadmap 中禁止使用强日期承诺，例如：

```text
7 月完成正式版
8 月上线商业版
9 月开放付费
```

应使用阶段性描述：

```text
当前阶段
下一阶段
长期探索
```

### 2.5 不做过度包装

该网站应保持克制、可信、真实。

禁止使用过度夸张表达，例如：

- 革命性游戏
- 重新定义互动叙事
- 史诗级大作
- 颠覆行业

允许使用客观表达，例如：

- 个人长期开发项目
- 早期试玩版
- 正在探索互动叙事和解谜机制
- 当前仍存在若干已知问题

## 5. 信息架构原则

### 5.1 首页优先传达四件事

首页访问者应在 10 秒内理解：

1. LoopTrain 是什么。
2. 当前项目处于什么阶段。
3. 如何进入试玩。
4. 哪里可以查看开发进展。

### 5.2 首页基本模块

首页建议包含：

- 顶部导航
- Hero 区域
- 当前版本状态
- 试玩入口按钮
- 最近开发日志
- 当前阶段路线摘要
- 已知问题摘要
- 页脚

### 5.3 试玩页面原则

`/play` 页面只承担一个核心目标：让用户进入游戏。

不要在试玩页面堆叠过多长文。

应包含：

- 当前版本号
- 早期试玩说明
- 推荐设备：手机竖屏
- 启动按钮
- 已知问题
- 简单反馈方式

### 5.4 开发日志原则

开发日志不是公众号长文。

日志应优先记录真实进展、真实问题和真实判断。

每篇日志建议包含：

```markdown
---
title: "日志标题"
date: "YYYY-MM-DD"
version: "v0.x.x"
status: "planning | doing | done | paused | cancelled"
tags:
  - 标签
summary: "一句话摘要"
---

## 背景

## 当前问题

## 本次调整

## 设计判断

## 后续计划

## 备注
```

## 7. 视觉风格要求

### 7.1 风格关键词

必须体现：

- 夜间列车
- 隧道感
- 时间循环
- 事故倒计时
- 低照度环境
- 冷色调屏幕光
- 轻微颗粒感
- 档案记录感
- 工程日志感

避免：

- 明亮卡通风
- 过度赛博朋克
- 高饱和霓虹色
- 二次元萌系官网风
- 商业游戏大宣发风
- 太多动效和花哨装饰

### 7.2 色彩系统

使用暗色主题作为默认主题。

推荐色彩变量：

```css
:root {
  --color-bg: #07090d;
  --color-bg-panel: #0d1118;
  --color-bg-elevated: #121722;
  --color-border: #263040;
  --color-border-soft: #1a2230;

  --color-text: #d7dde8;
  --color-text-muted: #8993a5;
  --color-text-subtle: #5f6b7d;

  --color-accent: #6cc7d9;
  --color-accent-soft: rgba(108, 199, 217, 0.14);
  --color-accent-strong: #9be8f4;

  --color-warning: #c98b55;
  --color-danger: #b84a4a;
  --color-danger-soft: rgba(184, 74, 74, 0.14);

  --color-success: #7ca982;
}
```

使用原则：

- 背景必须偏深，不使用纯黑大面积背景。
- 正文必须保证可读性。
- 强调色使用冷蓝、青色，不要大面积使用。
- 红色只用于危险、爆炸、倒计时、严重问题，不用于普通按钮。
- 页面整体对比度要足够，但不要刺眼。

### 7.3 字体要求

中文正文优先使用系统字体：

```css
font-family: system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", "Noto Sans SC", "Microsoft YaHei", sans-serif;
```

代码、版本号、日期、状态标签可使用等宽字体：

```css
font-family: "SFMono-Regular", Consolas, "Liberation Mono", Menlo, monospace;
```

要求：

- 不要依赖必须下载的商业字体。
- 不要把中文正文设为过细字重。
- 正文字号建议 16px 到 18px。
- 行高建议 1.7 到 1.9。
- 长文阅读区域宽度控制在 680px 到 760px。

## 8. 布局设计要求

### 8.1 移动端优先

LoopTrain 游戏本身是移动端竖屏体验，因此 Devlog 网站也必须移动端优先。

优先适配尺寸：

```text
390 × 844
430 × 932
768 × 1024
1440 × 900
```

### 8.2 页面宽度

建议：

```css
--layout-max: 1120px;
--content-max: 720px;
--content-wide: 960px;
```

规则：

- 首页可以使用较宽布局。
- 日志详情页必须使用窄阅读布局。
- 不允许桌面端正文铺满全屏。
- 移动端左右留白不得低于 20px。

### 8.3 顶部导航

导航应保持简洁。

建议导航项：

```text
首页 / 试玩 / 开发日志 / 路线图 / 更新记录 / 关于
```

移动端可以收起为菜单按钮。

导航风格：

- 半透明深色背景
- 细边框
- 轻微模糊可选
- 高度不要过大
- 当前页面需要有明确 active 状态

### 8.4 Hero 区域

Hero 应体现 LoopTrain 氛围，但不要压过内容。

建议元素：

- 大标题：LoopTrain Devlog / LoopTrain 开发日志
- 副标题：一款互动叙事解谜游戏的长期开发记录
- 简短说明
- 主按钮：开始试玩
- 次按钮：查看开发日志
- 当前版本 Badge
- 轻量背景纹理：轨道线、扫描线、隧道光、时间刻度

禁止：

- 大面积无意义背景图
- 影响文字阅读的强对比图片
- 太多动画
- 首屏加载过重

## 10. 动效要求

动效必须克制。

允许：

- 页面进入时轻微 opacity / translate
- hover 边框轻微亮起
- 按钮轻微位移
- 背景极低透明度扫描线
- 倒计时或版本状态的轻量闪烁

禁止：

- 大面积炫光
- 频繁闪烁
- 自动播放强动画
- 干扰阅读的背景运动
- 模拟故障效果过度使用

任何动效必须满足：

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation: none !important;
    transition: none !important;
  }
}
```

## 11. 内容风格要求

### 11.1 文案基调

文案应当：

- 冷静
- 克制
- 准确
- 真实
- 有开发记录感
- 不夸张
- 不煽情

适合表达：

```text
当前版本仍处于早期试玩阶段。
本次调整主要解决移动端遮罩错位问题。
音效系统暂不追求完整配乐，先覆盖关键交互节点。
```

不适合表达：

```text
这将是一次史诗级升级。
我们即将重新定义互动叙事。
玩家将体验前所未有的震撼。
```

### 11.2 开发日志写法

每篇日志应优先说明：

1. 发生了什么。
2. 为什么要做。
3. 当前问题是什么。
4. 做了什么判断。
5. 下一步是什么。

不要写成空泛感想。

### 11.3 版本记录写法

Changelog 必须具体。

推荐格式：

```markdown
## v0.4.3

日期：2026-06-xx

### 新增

- 增加移动端试玩入口。

### 修复

- 修复开场字幕在部分屏幕顶部露出的问题。

### 已知问题

- Server Plugin 未连接，状态无法完整持久化。
```

1. 先读取本 `AGENT.md`。
2. 不得引入数据库或登录系统，除非用户明确要求进入下一阶段。

## 25. 文档治理

Devlog 是 LoopTrain 的正式长期文档中心。新增或修改 `src/content/` 内容时，必须遵守：

- `TBD/` 只用于讨论稿，不代表当前事实。
- 正式长期文档最终进入 `devlog/src/content/`。
- 公开内容不得暴露核心悬疑谜底、隐藏身份或关键反转。
- 旧 ST / SillyTavern 内容只能作为历史资料出现，必须标记为历史或 legacy。
- Devlog 文章的 `date` 必须包含时间和时区，确保按时间排序稳定。
- 不得把计划、路线图或探索方向写成确定交付承诺。

内容或结构变更完成前运行：

```bash
python3 scripts/check_docs_governance.py
npm run build
npx astro check
```
