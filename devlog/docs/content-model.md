# LoopTrain Devlog 内容模型文档

版本：v0.1
用途：定义 LoopTrain 开发日志网站的信息架构、Markdown 内容模型、发布流程、编辑规范和内容策略。
受众：内容维护者、AI Coding Agent、前端开发者。

> 本文档是"设计文档"，不包含任何实际发布的日志正文。所有示例 Frontmatter 和内容仅为格式参考。

---

## 0. 网站首要目标：游戏试玩入口

LoopTrain Devlog 网站的**第一目标**是让访问者进入游戏试玩。

所有页面设计、内容编排、导航结构都必须服务于这个首要目标。开发日志、路线图、版本记录等模块是第二层目标——它们服务于"让项目长期可沉淀"，但不得挤占或模糊试玩入口。

### 核心约束

1. **首页必须在首屏提供醒目的试玩入口**。访问者应在 3 秒内看到"开始试玩"按钮，无需滚动。
2. **导航栏中"试玩"必须作为独立且醒目的条目**。不得将试玩入口隐藏在二级菜单或长文段落中。
3. **开发日志正文中不得将试玩链接作为次要信息**。如需在日志中提及试玩，必须使用显式链接，不得仅用内文提及。
4. **任何页面不得以开发日志、设计笔记等内容覆盖或推后试玩入口**。

## 1. 内容目录结构

所有可维护内容以 Markdown 文件形式存放在 `src/content/` 下。

```text
src/
├── data/
│   └── site-status.json     # 试玩入口元数据（阶段一必须，唯一数据源）
└── content/
    ├── devlog/              # 开发日志（阶段一必须）
    │   ├── 2026-06-13-why-devlog.md
    │   ├── 2026-06-13-current-issues.md
    │   └── 2026-06-13-audio-system-plan.md
    ├── characters/          # 角色档案（阶段一启用，初始占位）
    │   ├── xiaoning.md      # 小宁
    │   ├── zhao-chengjing.md  # 赵乘警
    │   └── shen-mohan.md    # 沈墨寒
    ├── design/              # 设计笔记（阶段二启用）
    │   └── .gitkeep
    ├── changelog/           # 版本更新记录（阶段一必须）
    │   └── v0.4.3.md
    └── pages/               # 独立页面内容（阶段一必须）
        ├── about.md         # 关于页面正文
        └── play.md          # 试玩页面说明正文（可选，也可直接在组件中维护）
```

### 目录说明

| 目录 | 阶段 | 说明 |
|------|------|------|
| `data/` | 阶段一 | **站点元数据**，静态 JSON，是试玩入口的唯一数据源 |
| `devlog/` | 阶段一 | 开发日志，每篇一个 `.md` 文件 |
| `characters/` | 阶段一 | **角色档案**，每个角色一个 `.md` 文件，仅包含公开安全信息 |
| `design/` | 阶段二 | 设计笔记，记录机制、叙事、技术决策 |
| `changelog/` | 阶段一 | 版本更新记录，按版本号命名 |
| `pages/` | 阶段一 | 独立页面长文内容（about 等） |

### 文件命名规则

- **开发日志**：`YYYY-MM-DD-slug.md`，其中 `slug` 为英文短标识，用于生成 URL。
  - 示例：`2026-06-13-why-devlog.md` → URL `/devlog/why-devlog`
- **版本记录**：`vX.Y.Z.md`，严格按语义化版本号命名。
  - 示例：`v0.4.3.md`
- **设计笔记**：`topic-slug.md`，不要求日期前缀（设计笔记主题相对稳定）。
  - 示例：`loop-mechanism.md`
- **角色档案**：`pinyin-slug.md`，使用角色名的拼音作为 slug，用于生成 URL。
  - 示例：`xiaoning.md` → URL `/characters/xiaoning`

## 1.1 站点状态数据文件：`src/data/site-status.json`

这是**整个网站的试玩入口元数据的唯一数据源**。首页、试玩页面、导航栏、版本徽章等所有位置引用试玩信息时，都从此文件读取，不得在其他位置硬编码。

所有字段支持静态 JSON，不依赖数据库，不依赖 CMS。

### Schema

```json
{
  "play": {
    "currentVersion": "v0.5-standalone-mvp",
    "playUrl": "/play/game",
    "playLabel": "进入游戏",
    "recommendedDevice": "手机竖屏",
    "stageStatus": "早期试玩版",
    "stageDescription": "当前版本仍处于早期测试阶段，部分角色、音效、界面和剧情逻辑尚未完成。",
    "knownIssues": [
      "首次进入可能仍显示 SillyTavern 原始界面。",
      "Server Plugin 未连接时，部分状态无法持久化。",
      "小宁立绘资源可能存在加载失败。",
      "移动端遮罩和开场字幕仍需优化。",
      "当前版本暂无完整音效系统。"
    ],
    "feedbackNote": "反馈方式：可以直接联系开发者，或在后续版本开放反馈表单。"
  },
  "site": {
    "lastUpdated": "2026-06-13",
    "currentPhase": "试玩版稳定"
  }
}
```

### 字段说明

| 字段路径 | 类型 | 必填 | 说明 |
|----------|------|------|------|
| `play.currentVersion` | string | 是 | 当前游戏版本号，格式 `vX.Y.Z` |
| `play.playUrl` | string | 是 | 试玩入口跳转目标 URL |
| `play.playLabel` | string | 是 | 试玩按钮文案 |
| `play.recommendedDevice` | string | 是 | 推荐设备说明 |
| `play.stageStatus` | string | 是 | 当前阶段状态标签 |
| `play.stageDescription` | string | 是 | 阶段说明文案，用于试玩页和首页状态卡片 |
| `play.knownIssues` | string[] | 是 | 当前已知问题列表 |
| `play.feedbackNote` | string | 是 | 反馈方式说明 |
| `site.lastUpdated` | string | 是 | 站点整体最后更新日期 |
| `site.currentPhase` | string | 是 | 当前路线图阶段名称 |

### 更新规则

1. 每次游戏版本变化时更新 `play.currentVersion` 和 `site.lastUpdated`。
2. 每次 Changelog 发布时同步更新 `play.knownIssues`（新增已解决问题、添加新发现问题）。
3. 当路线图阶段切换时更新 `site.currentPhase`。
4. `play.playUrl` 仅在游戏入口地址变化时修改。
5. 更新后需同时确认首页、`/play` 页、导航栏渲染正确。
6. **禁止**在此文件中编造进度、虚构已完成事项、或写入未实际部署的功能。

## 2. Frontmatter 规范

所有 Markdown 文件必须在文件顶部包含 YAML Frontmatter，用 `---` 包裹。

### 2.1 开发日志（devlog）

```yaml
---
title: "日志标题"
date: "YYYY-MM-DD"
version: "vX.Y.Z"
status: "planning"
tags:
  - 标签一
  - 标签二
summary: "一句话摘要，用于列表和卡片展示。"
---
```

#### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | string | 是 | 日志标题，使用中文 |
| `date` | string | 是 | 发布日期，格式 `YYYY-MM-DD` |
| `version` | string | 是 | 关联的游戏版本号，如 `v0.4.3` |
| `status` | enum | 是 | 当前状态，取值见 §3.1 |
| `tags` | string[] | 是 | 标签列表，中文，取值见 §3.2 |
| `summary` | string | 是 | 一句话摘要，不超过 80 字 |

### 2.2 设计笔记（design）

阶段二启用。Frontmatter 格式：

```yaml
---
title: "设计笔记标题"
date: "YYYY-MM-DD"
tags:
  - 标签
summary: "一句话摘要。"
spoiler: false
---
```

#### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | string | 是 | 笔记标题 |
| `date` | string | 是 | 创建或更新日期 |
| `tags` | string[] | 是 | 标签列表 |
| `summary` | string | 是 | 一句话摘要 |
| `spoiler` | boolean | 是 | 是否包含剧透内容。`true` 时应在页面做出醒目提示 |

### 2.3 版本记录（changelog）

```yaml
---
version: "v0.4.3"
date: "YYYY-MM-DD"
summary: "一句话描述本版本主要变化。"
---
```

#### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | string | 是 | 版本号 |
| `date` | string | 是 | 发布日期 |
| `summary` | string | 是 | 一句话版本摘要 |

### 2.4 独立页面（pages）

`about.md` 示例：

```yaml
---
title: "关于 LoopTrain"
updated: "YYYY-MM-DD"
---
```

`play.md` 示例（如采用 Markdown 管理）：

```yaml
---
title: "试玩入口"
version: "v0.4.3"
updated: "YYYY-MM-DD"
---
```

### 2.5 角色档案（characters）

```yaml
---
id: "xiaoning"
display_name: "小宁"
public_role: "列车乘客，对话角色"
availability: "public"
portrait_asset: "/images/characters/xiaoning.webp"
card_source_status: "ai_generated"
tags:
  - 公开角色
  - NPC
spoiler_level: "none"
summary: "小宁是 LoopTrain 试玩版中最早登场的对话角色之一。"
public_notes: "立绘由 AI 生成，素材授权状态待确认。"
related_devlog:
  - "2026-06-13-current-issues"
asset_license_status: "待确认"
---
```

#### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 角色唯一标识，拼音 slug，与文件名一致 |
| `display_name` | string | 是 | 角色公开中文名 |
| `public_role` | string | 是 | 公开身份/定位，不超过 20 字 |
| `availability` | enum | 是 | `public`（公开可展示）/ `locked`（仅占位，不展示详情） |
| `portrait_asset` | string | 否 | 立绘资源路径，相对于 `public/` |
| `card_source_status` | enum | 是 | 角色卡来源：`ai_generated` / `manual` / `mixed` / `待确认` |
| `tags` | string[] | 是 | 标签列表，取值见 §3.2 |
| `spoiler_level` | enum | 是 | 剧透等级，取值见 §3.4 |
| `summary` | string | 是 | 角色公开摘要，不超过 100 字 |
| `public_notes` | string | 否 | 公开补充说明，如素材状态、已知限制 |
| `related_devlog` | string[] | 否 | 关联的开发日志 slug 列表 |
| `asset_license_status` | string | 否 | 素材授权状态：`已确认` / `待确认` / `不可商用` |

#### availability 字段规则

- **`public`**：角色可公开展示。详情页渲染 Frontmatter + Markdown 正文。
- **`locked`**：角色存在但信息安全敏感，列表页仅显示一条占位条目，文字为 `display_name` + 锁定状态说明，**不**生成详情页路由。
- 当前唯一需要设为 `locked` 的角色：`小宁妈妈_隐藏节点`。其 `spoiler_level` 自动为 `critical`，所有其他字段在列表渲染时强制展示为"该角色信息暂不公开"。

## 3. 状态与标签体系

### 3.1 开发日志状态（status）

用于表示日志所记录事项的当前进展阶段。

| 值 | 中文 | 含义 | 适用场景 |
|----|------|------|----------|
| `idea` | 想法 | 初步想法，尚未进入规划 | 记录灵感、方向探索 |
| `planning` | 计划中 | 已确定要做，正在细化方案 | 方案设计、技术调研 |
| `doing` | 进行中 | 正在开发或实施 | 当前活跃的开发事项 |
| `done` | 已完成 | 已完成并验证 | 已合并、已上线 |
| `paused` | 暂停 | 因客观原因暂时搁置 | 依赖未就绪、优先级调整 |
| `cancelled` | 已取消 | 决定不再继续 | 方案废弃、方向调整 |

> 注意：Frontmatter 中 `status` 字段使用英文值（`idea` / `planning` / `doing` / `done` / `paused` / `cancelled`），页面渲染时显示对应的中文标签。

### 3.2 开发日志标签（tags）

标签采用中文，统一词汇，避免同义词混乱。

**当前阶段可用标签**：

| 标签 | 适用内容 |
|------|----------|
| `项目说明` | 项目方向、开发日志本身、建设目标 |
| `版本进展` | 版本发布、功能完成、里程碑 |
| `问题记录` | Bug、已知问题、缺陷整理 |
| `音效系统` | 音效、音乐、声音相关 |
| `UI优化` | 界面、遮罩、布局、交互 |
| `角色设计` | 角色、立绘、NPC |
| `剧情结构` | 叙事、分支、循环机制 |
| `技术实现` | 引擎、架构、工具链、部署 |
| `素材管理` | 图片、音效、字体来源与授权 |
| `设计思考` | 设计判断、取舍、方向讨论 |
| `移动端适配` | 竖屏、响应式、触屏交互 |
| `复盘` | 阶段总结、经验回顾 |
| `公开角色` | 角色档案中标记为可公开的角色 |
| `NPC` | 游戏中作为 NPC 出现的角色 |
| `隐藏角色` | 存在但信息敏感不公开的角色（配合 `locked` 状态使用） |

**标签使用规则**：

1. 每篇日志 1~4 个标签。
2. 优先使用已有标签。如需新增，先在本文档 §3.2 中登记，确保词汇统一。
3. 标签渲染时按上述顺序排列（非按添加顺序）。
4. 禁止英文标签与中文标签混用。

### 3.3 路线图状态（Roadmap Status）

用于表示路线图中各任务项的完成状态。

| 值 | 含义 | 视觉建议 |
|----|------|----------|
| `未开始` | 尚未启动 | 低亮、灰色 |
| `进行中` | 正在推进 | 弱高亮、青色 |
| `已完成` | 已完成并验证 | 弱绿 |
| `暂停` | 暂时搁置 | 弱橙 |
| `废弃` | 不再继续 | 弱红、删除线 |

路线图状态直接使用中文值，不设英文别名。

### 3.4 角色剧透等级（spoiler_level）

用于控制角色档案在网站上的可见内容范围。

| 值 | 含义 | 适用场景 | 渲染行为 |
|----|------|----------|----------|
| `none` | 无剧透风险 | 全部信息可安全展示 | 完整渲染所有字段和正文 |
| `mild` | 轻微剧透 | 涉及角色背景但非核心谜底 | 显示 Frontmatter 基础字段，正文渲染但顶部显示"本角色档案可能涉及轻微剧透"警告 |
| `critical` | 关键剧透 | 角色身份、行为与核心谜底强相关 | 不渲染详情页，列表仅显示占位条目；`availability` 应同时设为 `locked` |

> 注意：`spoiler_level` 由项目作者在撰写角色档案时判断赋值。如存疑，默认使用 `critical`，宁可少展示不可多暴露。

## 4. 正文结构规范

### 4.1 开发日志推荐结构

每篇开发日志正文建议包含以下章节：

```markdown
## 背景
（为什么需要做这件事、之前的状态）

## 当前问题
（具体问题描述，可列点）

## 本次调整
（做了什么、怎么做的）

## 设计判断
（为什么这样选择，放弃的方案，权衡考虑）

## 后续计划
（下一步做什么，不做什么）

## 备注
（可选：相关链接、参考资料、待确认事项）
```

### 4.2 版本记录推荐结构

```markdown
## vX.Y.Z

日期：YYYY-MM-DD

### 新增
- 项目一
- 项目二

### 修复
- 项目一
- 项目二

### 变更
- 项目一

### 已知问题
- 问题一：简述 + 当前状态
- 问题二：简述 + 当前状态
```

### 4.3 设计笔记推荐结构

```markdown
## 背景

## 设计目标

## 方案对比

## 最终决策

## 影响范围

## 后续观察
```

### 4.4 角色档案推荐结构

角色档案的 Markdown 正文为自由格式，但建议遵循以下安全结构：

```markdown
## 角色简介
（公开可见的身份信息、在游戏中的位置，1-2 段）

## 当前状态
（试玩版中的可用性、已完成的功能、已知问题，仅描述公开可见内容）

## 素材说明
（立绘来源、授权状态、是否 AI 生成、是否经过二次修改）

## 相关开发日志
（链接到相关的 devlog 条目）
```

**禁止在角色档案正文中写入**：
- 角色隐藏身份或真实动机
- 角色与其他角色的秘密关系
- 角色在核心谜题中的位置（属于问题还是答案）
- 尚未在游戏中通过正常流程可获知的角色信息
- 角色的未来剧情走向

**允许写入**：
- 角色的公开设定和可见身份
- 素材来源和授权信息
- 试玩版中的可用状态和已知问题
- 角色在游戏中的功能定位（如"对话角色"、"可交互 NPC"）

## 7. 剧透管理策略

### 7.1 核心原则

LoopTrain 是悬疑叙事解谜项目。公开网站**严禁**暴露以下内容：

- 最终事故真相
- 核心剧情反转
- 关键角色的隐藏身份
- 事故完整原因链
- 关键谜题的标准答案
- 尚未在游戏中公开的核心线索

### 7.2 允许公开的内容

以下内容可以安全地写入公开日志：

- 玩法机制说明（循环、对话、时间压力、AP 系统等）
- 设计方向和设计判断
- 技术实现方案
- 版本问题和已知缺陷
- 开发过程和进度
- 非核心世界观设定（列车设定、可见角色身份等）
- 素材来源和授权信息

### 7.3 设计笔记的剧透标记

设计笔记如涉及需要保护的剧情设计细节，必须在 Frontmatter 中将 `spoiler` 设为 `true`：

```yaml
spoiler: true
```

渲染时应在页面顶部显示剧透警告。具体哪些内容标记为剧透，由项目作者判断。

### 7.4 灰色地带处理

以下情况需谨慎处理：

- **NPC 行为模式**：可以说明 NPC 有自主时间线，但不应透露具体行为触发条件。
- **失败结算**：可以说"失败后保留部分线索"，但不应透露哪些线索会保留。
- **结局数量**：在长期阶段可以提及"计划多个结局"，但不应透露结局内容。
- **隐藏角色**：可以提及存在隐藏 NPC，但不应透露身份或触发条件。

### 7.5 角色档案专项剧透规则

角色档案是最容易产生剧透风险的内容类型之一。以下规则优先于 §4.4 和 §7.1-§7.4：

1. **禁止以"角色介绍"为名暴露谜底**。角色档案只能描述试玩版中正常流程可见的信息。
2. **禁止描述角色之间的关系网络**。即使两个角色在试玩版中都有出现，也不应在角色档案中说明他们之间的秘密关联。
3. **禁止在角色档案中标注"重要角色"、"关键角色"、"最终 Boss"等剧透性标签**。只能使用中性标签如 `公开角色`、`NPC`、`隐藏角色`。
4. **禁止在角色档案中写"真实身份是……"**。`public_role` 字段只能描述表面可见身份。
5. **`spoiler_level: critical` 的角色**：不创建详情页，不渲染正文，列表仅显示占位条目。唯一例外是项目作者在本地维护的非公开版本可保留完整内容，但该文件不得提交到公开仓库。
6. **`spoiler_level: mild` 的角色**：详情页顶部必须渲染剧透警告横幅。正文中涉及背景信息的部分需要作者逐段判断是否过度暴露。
7. **隐藏角色的存在本身可以提及**（如路线图中"完成 3 个 NPC + 1 个隐藏 NPC 的试玩链路"），但不得在角色档案栏目中展开任何描述。

## 8. 路线图状态模型

### 8.1 阶段结构

路线图按"阶段"组织，每个阶段包含多个"任务项"。

```text
阶段（Phase）
├── 阶段名称：试玩版稳定
├── 阶段状态：进行中
├── 阶段说明：简要描述阶段目标
└── 任务项（Item）
    ├── 任务名称
    ├── 状态：未开始 / 进行中 / 已完成 / 暂停 / 废弃
    ├── 优先级：高 / 中 / 低
    └── 说明：简短描述
```

### 8.2 状态流转规则

```
未开始 ──→ 进行中 ──→ 已完成
  │          │
  └──────────┼──→ 暂停 ──→ 进行中（恢复）
             │
             └──→ 废弃
```

- **未开始 → 进行中**：开始实际推进
- **进行中 → 已完成**：达到验收标准
- **进行中 → 暂停**：因依赖未就绪、优先级调整等原因暂缓
- **进行中 → 废弃**：决策变更，不再继续
- **暂停 → 进行中**：条件满足，恢复推进
- **已完成**：终态，不再流转

### 8.3 路线图不包含的内容

- 具体完成日期或时间窗口
- 版本号与路线图任务的一一绑定（只说阶段，不说"v0.5 之前完成"）
- 过于细节的子任务（路线图保持宏观视角，细节写在开发日志中）

## 9. 素材授权记录模型

阶段二启用 `/assets` 页面时，采用以下记录模型。

### 9.1 记录字段

| 字段 | 类型 | 说明 |
|------|------|------|

5. 音效、字体、图片、立绘全部纳入记录范围。

## 10. 编辑体例与文案规范

### 10.1 文案基调

所有对外文案（首页、日志、版本记录、关于页面）必须遵循以下基调：

- **冷静**：不煽情、不夸张、不制造紧迫感
- **克制**：不过度修饰，不堆叠形容词
- **准确**：技术术语使用正确，不做模糊描述
- **真实**：只写实际发生的事，不编造进展
- **记录感**：像开发日志，不像营销文案

### 10.2 适合的表达方式

```text
当前版本仍处于早期试玩阶段。
本次调整主要解决移动端遮罩错位问题。
音效系统暂不追求完整配乐，先覆盖关键交互节点。
目前已知立绘资源存在加载失败的情况，下一步进行排查。
```

### 10.3 禁止的表达方式

```text
这将是一次史诗级升级。
我们即将重新定义互动叙事。
玩家将体验前所未有的震撼。
革命性游戏体验。
颠覆行业的创新。
```

### 10.4 标点与格式

- 中文正文使用全角标点。
- 英文和数字与中文之间加空格：`v0.4.3 版本`、`3 个 NPC`。
- 版本号使用等宽字体渲染，但在 Markdown 源码中写作普通文本。
- 代码、路径、文件名使用反引号包裹：`enableServerPlugins`、`src/content/devlog/`。
- 不使用 emoji 作为正文标点或装饰（不在文章中用 emoji 替代项目符号或表达语气）。
- 不使用波浪号（`~`）或连续感叹号（`!!`）表达语气。

### 10.5 人称与视角

- 使用第一人称（"我"）或中性描述（"本项目"、"当前版本"）。
- 不使用"我们"来暗示团队（LoopTrain 是个人项目）。
- 不虚构"玩家反馈"、"用户表示"、"大家觉得"。除非有真实的、可引用的反馈来源。

### 10.6 标题与摘要

- 标题使用陈述句，不使用问句或感叹句。
  - 适合：`准备增加音效系统`
  - 不适合：`音效系统要来了！` 或 `我们为什么还要做音效？`
- 摘要必须概括正文核心信息，而非用问句吊胃口。
- 标题中不使用英文（专有名词如 LoopTrain、SillyTavern 除外）。

### 10.7 版本号书写

- 所有版本号以 `v` 开头：`v0.4.3`
- 不要写 `0.4.3` 或 `V0.4.3`
- 语义：`v主版本.功能版本.修复版本`
  - 主版本：完整游戏或重大架构变化
  - 功能版本：新增玩法、系统、页面
  - 修复版本：Bug 修复、体验优化、素材调整

## 11. 页面内容模型（首页与关键页面）

### 11.1 首页数据源与模块顺序

首页模块必须按以下**强制顺序**渲染，不可调整：

```
1. 顶部导航（含"试玩"入口）
2. Hero 区域
3. 试玩入口主区块  ← 首屏必须可见，在开发日志之前
4. 当前版本状态卡片
5. 最近开发日志  ← 在试玩入口之后
6. 路线图摘要
7. 页脚
```

首页动态内容来自以下数据源：

| 内容区域 | 数据来源 | 说明 |
|----------|----------|------|
| 试玩入口主区块 | `src/data/site-status.json` → `play.*` | 版本号、设备建议、阶段说明、"进入游戏"按钮 |
| 当前版本状态 | `src/data/site-status.json` → `play.*` + `site.*` | 当前版本、阶段、更新日期、已知问题数量、下一步重点 |
| Hero 区域 | 静态文案 + site-status | 标题、副标题、简介文案、主按钮（开始试玩）、次按钮（查看开发日志） |
| 最近日志 | `devlog/` 目录下最近 3 篇（按 date 排序） | 显示标题、日期、摘要、标签 |
| 路线图摘要 | Roadmap 数据源 | 当前阶段目标简述 |

### 11.2 试玩页面数据源

`/play` 页面**只承担一个核心目标：让用户进入游戏**。不要在试玩页面堆叠过多长文。

| 内容区域 | 数据来源 |
|----------|----------|
| 版本号、设备建议、阶段说明 | `src/data/site-status.json` → `play.*` |
| 进入游戏按钮（含跳转 URL） | `src/data/site-status.json` → `play.playUrl` + `play.playLabel` |
| 已知问题列表 | `src/data/site-status.json` → `play.knownIssues` |
| 反馈方式 | `src/data/site-status.json` → `play.feedbackNote` |
| 试玩说明正文（可选） | `pages/play.md` 或组件内维护 |

页面布局原则：
- 游戏标题 + 版本 Badge 在最顶部。
- "进入游戏"按钮紧随其后，**不**需要用户先阅读长文。
- 已知问题折叠或置于按钮下方，不作为进入游戏的障碍。
- 开发日志链接、路线图链接等为次要链接，置于页面底部。

### 11.3 开发日志列表页

- 所有 `devlog/` 下文件按 `date` 倒序排列
- 每篇显示：标题、日期、标签、摘要、状态
- 无分页（首批内容量不足以需要分页）

### 11.4 开发日志详情页

- 路由：`/devlog/:slug`，slug 取自文件名 `YYYY-MM-DD-slug.md` 中的 `slug` 部分
- 页面渲染：Frontmatter + Markdown 正文
- 底部可显示"上一篇 / 下一篇"导航

### 11.5 角色档案列表页与详情页

#### 路由选择：`/characters`

选用 `/characters` 而非 `/character-cards`，理由：

- 该页面向所有访问者（试玩用户、潜在合作者），而非仅面向 ST 技术用户。
- "角色"比"角色卡"更通用、更直观，不暗示 SillyTavern 特定格式。
- `/character-cards` 暗示"卡片"格式，可能让访问者期待可下载的 ST 角色卡文件，而当前阶段不承诺提供此功能。
- 如需未来提供 ST `.STcard.png` 下载，应放在独立子路径（如 `/characters/cards/`），不与角色介绍页混淆。

#### 角色列表页 `/characters`

- 遍历 `src/content/characters/` 下所有 `availability: public` 的 `.md` 文件
- 每篇显示：`display_name`、`public_role`、`summary`、`tags`、`portrait_asset`（缩略图）
- 按 `display_name` 中文拼音排序
- 列表末尾可追加 locked-entries 占位行（如"小宁妈妈"）
- **不**展示 `spoiler_level` 字段（避免通过等级反向推断敏感度）
- 页面上方可有简短栏目说明："以下为 LoopTrain 试玩版中已登场的角色简介。部分角色信息暂不公开。"

#### 角色详情页 `/characters/:id`

- 仅 `availability: public` 且 `spoiler_level` 为 `none` 或 `mild` 的角色生成详情页
- `spoiler_level: mild` 的页面顶部渲染剧透警告横幅
- 页面渲染顺序：立绘 → 角色名 → 公开身份 → 标签 → 正文
- 正文下方可显示"相关开发日志"链接列表
- `availability: locked` 的角色**不**生成详情页路由

#### 角色档案的导航定位

- 导航栏新增"角色"条目，位于"开发日志"之后、"路线图"之前。
- 角色页面不是网站第一目标（见 §0），不可在首页将角色列表置于试玩入口之前。
- 角色栏目的定位是"帮助访问者了解游戏中出现的角色"，而不是替代试玩体验。

## 12. 后续扩展预留

### 12.1 阶段二新增内容类型

当 `/design` 和 `/assets` 页面启用时，对应的内容模型已在本文档中预留（§2.2、§4.3、§9），无需重新设计。

角色档案（`/characters`）已在阶段一启用基础框架和首批占位条目（§2.5、§5.6）。阶段二可扩展：
- 角色正文从占位扩展为完整简介
- 增加角色立绘展示
- 增加角色相关的设计笔记链接

### 12.2 标签筛选

当前首批内容量不足以需要标签筛选功能。当 `devlog/` 目录下文件超过 15 篇时，可考虑在 `/devlog` 列表页增加标签筛选。

### 12.3 RSS

当开发日志达到一定更新频率（例如每月至少 2 篇、持续 3 个月以上）时，可考虑生成 RSS feed。RSS 内容直接从 `devlog/` Frontmatter 提取。

### 12.4 搜索

当前阶段不做全文搜索。如需启用，建议使用客户端搜索方案（如 Pagefind），不引入后端搜索服务。
