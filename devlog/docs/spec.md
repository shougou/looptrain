# LoopTrain 开发日志网站 Spec 文档

版本：v0.1
项目名称：LoopTrain Devlog / LoopTrain 开发日志
域名：looptrain.me
项目性质：个人长期开发项目展示站、开发日志站、版本记录站
核心目标：记录 LoopTrain 项目的开发进展、设计思考、路线计划与试玩入口。

---

# 1. 项目背景

LoopTrain 是一个个人长期开发的互动叙事解谜游戏项目。

当前游戏处于早期试玩阶段，已经具备基本方向：

* 移动端竖屏体验
* 列车循环题材
* 对话解谜玩法
* 时间压力机制
* NPC 自主时间线
* SillyTavern 作为底层对话引擎
* LoopTrain 控制层负责游戏 UI、状态、流程与叙事逻辑
* 当前已有公网试玩入口
* 后续计划增强音效系统、角色系统、剧情结构、失败结算、素材管理等内容

由于项目是个人长期兴趣项目，不适合一开始做成复杂官网或商业化宣传站。更合理的方向是建立一个轻量、可持续维护的开发日志网站，用于沉淀项目进展、设计决策和版本记录。

# 2. 网站定位

## 2.1 网站名称

中文名称：

> LoopTrain 开发日志

英文名称：

> LoopTrain Devlog

## 2.2 网站一句话定位

> 一款互动叙事解谜游戏的长期开发记录。

## 2.3 网站说明

LoopTrain 开发日志用于记录 LoopTrain 游戏项目从早期 Demo 到完整作品的演化过程，包括试玩入口、版本更新、开发日志、路线计划、设计笔记和素材说明。

该网站不是传统意义上的商业游戏官网，也不是社区平台，而是一个面向个人长期创作的项目档案站。

# 3. 建设目标

## 3.1 核心目标

1. 提供稳定的游戏试玩入口。
2. 记录 LoopTrain 项目的版本演进。
3. 沉淀开发过程中的设计思考。
4. 公开项目计划和当前进度。
5. 为未来测试用户、合作者、读者提供低成本了解项目的入口。
6. 帮助开发者持续推进项目，而不是只停留在本地实验阶段。

## 3.2 非目标

当前阶段不做以下内容：

1. 不做复杂游戏官网。
2. 不做用户注册和登录。
3. 不做论坛、评论区、社区系统。
4. 不做付费系统。
5. 不做复杂后台 CMS。
6. 不做多人协作管理系统。
7. 不公开核心悬疑谜底。
8. 不承诺明确上线日期。
9. 不做过度宣传包装。

## 6.2 试玩页面 `/play`

### 页面目标

让用户直接进入 LoopTrain 试玩版。

### 页面内容

#### 1. 游戏标题

```text
LoopTrain 试玩版
```

#### 2. 版本提示

```text
当前版本：v0.5-standalone-mvp
推荐设备：手机竖屏
状态：早期试玩版
```

#### 3. 试玩说明

文案示例：

> 当前版本仍处于早期测试阶段，部分角色、音效、界面和剧情逻辑尚未完成。建议使用手机竖屏访问。

按钮文案：

点击后跳转到当前游戏入口，例如：

```text
/play/game
```

或者嵌入现有游戏页面。

示例：

* 首次进入可能仍显示 SillyTavern 原始界面。

* 小宁立绘资源可能存在加载失败。
* 移动端遮罩和开场字幕仍需优化。
* 当前版本暂无完整音效系统。

#### 6. 反馈入口

第一阶段可以只放邮箱或文字说明，不做表单。

示例：


## 6.4 开发日志详情 `/devlog/:slug`

### 页面目标

展示单篇开发日志完整内容。

### 页面结构

每篇日志建议使用统一结构：

```markdown
# 标题

日期：
版本：
标签：
状态：

## 背景

## 当前问题

## 本次调整

## 设计判断

## 后续计划

## 备注
```

### 内容原则

1. 不追求长文。
2. 不需要每篇都像公众号文章。
3. 重点记录真实进展和真实问题。
4. 可以记录失败、返工、设计变化。
5. 不公开核心悬疑谜底。
6. 不把计划写成承诺。

## 6.5 路线图 `/roadmap`

### 页面目标

说明项目当前阶段、下一阶段和长期方向。

### 页面结构

路线图建议分为三层：

---

### 当前阶段：试玩版稳定

目标：

* 隐藏 SillyTavern 原始界面
* 修复移动端遮罩问题
* 修复小宁立绘资源 404
* 梳理 LT 控制层和 ST 引擎层边界
* 接入基础音效系统
* 完成基础失败结算卡
* 完成 3 个 NPC + 1 个隐藏 NPC 的试玩链路
* 优化开场字幕和移动端竖屏表现

---

### 下一阶段：体验增强

目标：

* 增加背景音乐
* 增加关键事件音效
* 增加循环重启音效
* 完善 NPC 自主时间线
* 增强角色对话状态
* 完善失败后的线索继承
* 增强对话与行动切换体验
* 增加剧情节点配置能力

---

### 长期阶段：完整游戏

目标：

* 扩展至 10 个以上 NPC
* 完成事故真相主线
* 完成多个结局
* 完成角色卡、世界书和剧情节点管理机制
* 建立用户游玩数据统计
* 完成完整音效和音乐设计
* 建立素材版权记录
* 探索独立游戏发布方式

---

### 路线图原则

1. 使用“当前 / 下一步 / 长期探索”，不使用强日期承诺。
2. 明确当前优先级。
3. 每完成一项后更新状态。
4. 保留历史路线图变更记录。

## 6.6 更新记录 `/changelog`

### 页面目标

以版本为单位记录项目变化。

### 内容格式

```markdown
# Changelog

## v0.4.3

日期：2026-06-xx

### 新增

- 增加公网试玩入口。
- 增加移动端基础界面。
- 增加 DeepSeek 模型切换入口。

### 修复

- 修复部分交互状态异常。

### 已知问题

- Server Plugin 未连接。
- 小宁立绘资源 404。
- 首次进入仍可能显示 SillyTavern 原始界面。
- 移动端开场遮罩仍需调整。
```

### 版本命名规则

建议采用语义化版本的简化形式：

```text
v主版本.功能版本.修复版本
```

示例：

```text
v0.4.3
```

规则：

* 主版本：完整游戏或重大架构变化
* 功能版本：新增玩法、系统、页面
* 修复版本：bug 修复、体验优化、素材调整

## 6.7 设计笔记 `/design`

### 页面目标

沉淀重要设计思考。

### 适合记录的内容

* 为什么采用列车循环结构
* 为什么使用移动端竖屏
* 为什么使用 SillyTavern 作为底层引擎
* 对话和行动如何切换
* 时间压力如何设计
* NPC 自主时间线如何影响玩家
* 失败结算如何保留线索
* 音效系统如何增强沉浸感
* 如何避免核心悬疑提前剧透

### 注意事项

设计笔记可以公开设计方法，但不应公开最终谜底。

可以写：

```text
本项目希望让玩家通过多轮循环逐步修正判断，而不是一次性获得答案。
```

不建议写：

```text
最终真相是某某角色导致事故。
```

## 6.8 素材记录 `/assets`

### 页面目标

记录项目使用的素材来源、授权和用途。

### 内容字段

建议记录：

```text
素材名称
素材类型
用途
来源
授权方式
是否可商用
是否需要署名
是否经过 AI 生成
是否经过二次修改
备注
```

### 示例

| 素材名称              | 类型 | 用途    | 来源    | 授权   | 是否可商用 | 备注    |
| ----------------- | -- | ----- | ----- | ---- | ----- | ----- |
| train_ambience_01 | 音效 | 列车环境声 | 待定    | 待确认  | 待确认   | 第一版占位 |
| xiaoning_portrait | 立绘 | 小宁角色  | AI 生成 | 自有生成 | 待确认   | 需统一风格 |

### 原则

1. 所有公开发布素材必须记录来源。
2. 不使用授权不明的商用素材。
3. AI 生成素材也需要记录生成方式和用途。
4. 音效、字体、图片、立绘都要纳入记录。

# 7. 内容管理规格

## 7.1 内容形式

第一阶段建议全部使用 Markdown 文件维护。

不引入数据库，不做复杂后台。

推荐内容目录：

```text
content/
├── devlog/
│   ├── 2026-06-13-audio-system-plan.md
│   └── 2026-06-xx-mobile-ui-fix.md
├── design/
│   ├── loop-mechanism.md
│   └── npc-timeline.md
├── changelog/
│   └── v0.4.3.md
└── pages/
    ├── about.md
    └── roadmap.md
```

## 7.2 Markdown Frontmatter

开发日志文件格式：

```markdown
---
title: "准备增加音效系统"
date: "2026-06-13"
version: "v0.4.3"
status: "planning"
tags:
  - 音效系统
  - 体验优化
summary: "当前游戏完全没有声音，下一步需要建立最小可用音效系统。"
---

正文内容……
```

## 7.3 状态字段

建议支持：

```text
idea        想法
planning    计划中
doing       进行中
done        已完成
paused      暂停
cancelled   已取消
```

## 7.4 标签字段

建议统一使用中文标签，避免中英文混用。

# 9. 技术架构建议

## 9.1 第一阶段推荐方案

推荐使用静态站点方案。

可选技术栈：

```text
Next.js / Astro / VitePress / Nuxt Content
```

从维护成本看，推荐顺序：

1. Astro
2. VitePress
3. Next.js

## 9.2 推荐架构

如果目标是长期开发日志站，推荐：

```text
Astro + Markdown + 静态生成 + Nginx
```

理由：

* 适合内容站
* Markdown 维护简单
* 性能好
* 部署简单
* 不需要数据库
* 不需要复杂后端
* 适合个人长期维护

## 9.3 部署方式

部署环境：

```text
云主机
Nginx
Node.js
PM2 可选
HTTPS 证书
```

推荐部署方式：

```text
本地构建静态文件
上传 dist 到服务器
Nginx 指向 dist 目录
```

也可以使用 GitHub Actions 自动部署，但第一阶段不是必须。

# 13. 安全与隐私要求

1. 不采集用户个人信息。
2. 不开放用户上传。
3. 不开放评论区。
4. 不保存用户输入。
5. 不暴露服务器敏感路径。
6. 不把游戏后台配置、API Key、模型 Key 写入前端。
7. 所有 HTTPS 访问。
8. Nginx 禁止访问隐藏文件，例如 `.env`、`.git`。
9. 日志中避免记录敏感信息。

# 16. 验收标准

## 16.1 功能验收

1. 访问 `looptrain.me` 可以看到首页。
2. 首页可以清楚说明 LoopTrain 是什么。
3. 首页有明显的试玩入口。
4. `/play` 页面可以进入游戏试玩。
5. `/devlog` 可以看到开发日志列表。
6. 单篇开发日志可以正常打开。
7. `/roadmap` 可以看到当前阶段和后续计划。
8. `/changelog` 可以看到版本记录。
9. `/about` 可以看到项目介绍。
10. 所有页面移动端可正常阅读。

## 16.2 内容验收

1. 至少有 3 篇开发日志。
2. 至少有 1 条版本记录。
3. Roadmap 包含当前阶段、下一阶段、长期阶段。
4. 首页显示当前版本号。
5. 已知问题明确列出。
6. 不公开核心剧情谜底。
7. 不出现无法兑现的时间承诺。

## 16.3 技术验收

1. 页面首屏加载速度可接受。
2. 静态资源路径正确。
3. 无明显 404。
4. HTTPS 正常。
5. Nginx 配置正确。
6. 不暴露 `.env`、`.git` 等敏感文件。
7. 移动端布局无明显错位。
8. Markdown 内容可以正常渲染。

# 17. 首批内容建议

## 17.1 第一篇开发日志

标题：

```text
为什么要建立 LoopTrain 开发日志
```

摘要：

```text
LoopTrain 是一个个人长期开发的互动叙事解谜游戏项目。建立开发日志的目的不是宣传，而是持续记录项目演化、设计判断和版本进展。
```

## 17.2 第二篇开发日志

标题：

```text
当前试玩版的问题整理
```

内容要点：

* SillyTavern 原始界面仍可能显示
* 小宁立绘资源 404
* 移动端遮罩存在问题
* 暂无声音系统
* Server Plugin 未连接
* NPC 和剧情链路还不完整

## 17.3 第三篇开发日志

标题：

```text
准备增加音效系统
```

内容要点：

* 当前游戏完全没有声音
* 声音会直接影响沉浸感
* 第一阶段不追求完整配乐
* 先做最小可用音效系统
* 优先覆盖按钮、对话、倒计时、爆炸、循环重启、线索发现

# 20. 总结

LoopTrain 开发日志网站的核心价值不是包装项目，而是让项目具备长期沉淀能力。

第一版应保持简单、稳定、可维护：

```text
能试玩
能记录
能更新
能回顾
能长期维护
```

这是当前阶段最合理的网站形态。
